# 実装計画 - ホットプロファイル連携コードの改善

## ゴール
現在のホットプロファイル連携コードにおいて、定義されていない関数が存在し、実行時にエラーになる問題があります。また、バッチ処理内での同期的なAPI呼び出しによるパフォーマンス低下も懸念されます。
本計画では、これらの未定義関数を実装し、データ取得プロセスを最適化して、タイムアウトやレート制限のリスクを低減することを目的とします。

## 特定された重要な問題
- **未定義関数**: 以下の関数が呼び出されていますが、読み込んだファイル内に定義が見つかりません。
    - `fetchLatestVisitForClient` (`企業情報.js` で使用)
    - `fetchCompanyPage` (`不足分の再取得.js` で使用)
    - `fetchCompanyById` (`企業情報.js` で単一更新に使用)
- **パフォーマンス**: `upsertClients` 関数内で、`enableLastTouch` が有効な場合、各クライアントごとに同期的に `fetchLatestVisitForClient` を呼び出しています。これは大量データの処理時に著しくパフォーマンスを低下させます。

## 変更案

### 1. 未定義関数の実装
文脈と既存のロジックに基づいて、これらの関数を再作成します。

#### `共通.js`
- **[新規関数]** `fetchLatestVisitForClient(apiKey, clientName, clientId)`:
    - `startLastTouchStep` と同様のロジックを、単一クライアント向けに実装します。
    - *注意*: これをループ内で大量に呼び出すことは推奨されません。

#### `企業情報.js`
- **[新規関数]** `fetchCompanyPage(companyId, ...)`:
    - 特定ページの企業データを取得するロジックを実装します (`refetch` ロジックで使用)。
- **[新規関数]** `fetchCompanyById(companyId)`:
    - ID指定で単一の企業情報を取得するロジックを実装します。

### 2. `upsertClients` の最適化
- **[変更]** `企業情報.js` 内の `upsertClients`:
    - `fetchLatestVisitForClient` の同期呼び出しを削除します。
    - または、`clients.length` が少ない場合（例: 5件未満）のみ実行するように制限し、一括更新処理をブロックしないようにします。
    - 基本的に「ラストタッチ」情報の更新は、並列処理に対応した `updateLastTouchStep` トリガーに委譲すべきです。

### 3. 取得ログの整理
- **[変更]** `ensureLogSheet`, `appendLogEntries`, `appendRunLog` (`企業情報.js`):
    - ログ項目を簡素化し、ユーザーにとって重要な情報のみに絞ります。
    - **削除予定の列**: `from`, `to`, `page` (デバッグ用のため), `レコードNo`, `テナント顧客ID` (マスタ情報のためログには不要と判断)
    - **維持/整理後の列構成案**: `["日時", "操作", "会社ID", "会社名", "更新日時", "進捗状況", "備考"]`
    - **進捗状況列の新設**: `cursor=2200` のような内部値ではなく、`「2200件完了 (今回: 200件)」` のような分かりやすい情報を専用列に出力します。
    - 備考にはその他の補足情報やエラー詳細を出力します。

### 4. 通知精度の向上 (複数トリガー対応)
- **[変更]** `startWeeklyStep` / `startSingleIdStep`:
    - 処理開始時に集計用プロパティ (`STATS_START_TIME`, `STATS_ADDED`, `STATS_UPDATED`, `STATS_MISS`) を初期化します。
- **[変更]** `updateCompanyListStepSingleIdMode`:
    - 各バッチ実行の最後に、その回の処理件数（追加・更新・見つからない）をプロパティに加算（累積）します。
    - 全バッチ完了時に、プロパティに保存された「開始時刻」と「累積件数」を使用して、正確な所要時間と総件数を通知します。
    - 完了後にプロパティを削除します。

### 5. クリーンアップ
- すべてのAPI呼び出しで `getApiKey()` を使用し、エラーを適切にハンドリングするように統一します。

## 検証計画
GASプロジェクトであるため、以下の手順で検証を行います。
1.  **コードレビュー**: 実装ロジックが HotProfile API の仕様に適合しているか確認。
2.  **実行テスト**: ユーザー側で `startSingleIdStep` や `startLastTouchStep` トリガーを実行し、動作を確認してもらう。
3.  **ログ確認**: 実行後に `取得ログ` シートを確認し、「MISS」や「ERROR」が発生していないか検証する。
