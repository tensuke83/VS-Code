# レコード番号（テナントID）による重複排除とマッチング精度の向上

## 課題
同一会社IDで「テナント_顧客ID」が重複している（または空欄で重複している）場合、既存の実装では「会社ID_テナント顧客ID」をキーとして重複チェックや行特定を行っているため、キーの衝突が発生する。
その結果、以下の問題が起きている：
1. `processedIds` で重複判定され、2件目以降の処理がスキップされる（値が入らない）。
2. `existingDataMap` でキーが重複し、マップ上で1つの行情報のみが残り、複数のテナントデータが同一の行を更新してしまう（または片方が更新されない）。

## 解決策
テナントマスタの「id」列（= `recordNo`）はユニークであるため、これを主キーとして利用する。

### 1. `企業情報.js` / `buildExistingDataMapByHeader`
- 「レコード番号」列を読み込む。
- マップへの登録ロジックを変更する：
  - **レコード番号がある行**: `REC:{recordNo}` をキーとしてマップに登録する。CompositeKeyでの登録は行わない（キー重複による誤マッチングを防ぐため）。
  - **レコード番号がない行**: 従来通り `CompositeKey` で登録する（過去データ救済）。

### 2. `企業情報.js` / `upsertClients`
- `processedIds` の重複チェックキーを変更する：
  - 単なる `CompositeKey` ではなく、ユニーク性を保証するため `PROC:{responseId}_{recordNo}` を使用する。
  - `recordNo` がない場合は `CompositeKey` を使う。
- 既存行の検索ロジック (`existing`) を変更する：
  - まず `REC:{recordNo}` でマップを検索する。
  - ヒットしなければ、フォールバックとして `CompositeKey` で検索する。

## メリット
- テナント顧客IDが空欄であっても、テナントマスタ上で別レコードであれば、`recordNo` によって明確に区別して処理・更新できる。
- 既存の「レコード番号」列が入っていないデータに対しても、フォールバックで対応可能。

## 実装ステップ
1. `buildExistingDataMapByHeader` で「レコード番号」列を取得し、マップ登録ロジックを分岐させる。
2. `upsertClients` で `processedIds` のキー生成ロジックを変更する。
3. `upsertClients` で `existingDataMap` の検索ロジックを変更する。
